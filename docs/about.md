## 1. Функциональные требования

### 1.1. Роли

- Ведущий (Host) — создаёт квизы, запускает сессии, управляет ходом игры, видит результаты в реальном времени
- Участник (Player) — подключается по коду сессии, отвечает на вопросы, видит свой счёт

### 1.2. Квизы

- Квиз — набор вопросов (название, описание, порядок)
- Вопрос — текст, тип ответа (один/несколько вариантов), варианты ответов, правильный ответ(ы), время на ответ (обязательно)
- Квизы всегда сохраняются на сервере — можно запускать многократно без пересоздания
- Ведущий создаёт квиз и получает уникальный код квиза для повторного использования
- Квизы хранятся по коду (без привязки к пользователям)
- Валидация структуры на сервере

### 1.3. Сессия игры

- Сессия привязана к квизу и ведущему, имеет уникальный PIN-код (6 цифр)
- Состояния: Lobby → Running → Finished
- Участники подключаются по PIN в состоянии Lobby
- Ведущий запускает игру (Lobby → Running), новые участники не принимаются
- Таймер на вопрос обязателен и должен быть максимально синхронным для всех участников — сервер является источником истины времени
  - Сервер хранит дедлайн таймера и периодически рассылает оставшееся время всем участникам одновременно
  - Клиенты не считают время самостоятельно, а синхронизируются по серверным событиям `TimerUpdate`
  - При старте вопроса сервер рассылает `QuestionStarted` с временем старта и длительностью
  - При истечении времени — атомарная рассылка `QuestionTimeout` всем участникам
- Ведущий может принудительно перейти к следующему вопросу
- По окончании всех вопросов — переход в Finished, формирование таблицы лидеров

### 1.4. Ответы и очки

- Участник отправляет один ответ на вопрос
- Очки: правильность + скорость ответа (быстрее = больше очков)
- Накопление очков, итоговый рейтинг по убыванию

### 1.5. Синхронизация и уведомления

- Сервер рассылает события: новый вопрос, окончание таймера, результаты, конец игры
- Участники видят вопрос синхронно (сервер рассылает одновременно)
- Синхронизация таймера: сервер рассылает `TimerUpdate` с оставшимся временем периодически (каждые 50-100мс) всем участникам одновременно
- Клиенты обновляют UI таймера по серверным значениям, игнорируя локальный отсчёт
- Ведущий видит: кто подключился, кто ответил, предварительные результаты

### 1.6. Многопоточность

Сервер:
- Acceptor Thread (1) — принимает новые соединения
- Worker Threads (N) — обрабатывают I/O соединений, команды клиентов
- Timer Thread (1) — периодически обновляет таймеры сессий, рассылает TimerUpdate
- `SessionManager` и `QuizRegistry` — потокобезопасны (мьютексы)
- Broadcast событий — атомарная рассылка всем участникам сессии

Клиент:
- Main Thread (GUI) — обработка UI, все операции с виджетами только здесь
- Network Thread — управление TCP-соединением, парсинг протокола
- Таймер?
- Взаимодействие потоков через сигналы Qt 

### 1.7. Надёжность

- Корректная обработка отключений участников и ведущего
- Heartbeat для обнаружения разрывов соединения
- Валидация входящих данных на сервере
